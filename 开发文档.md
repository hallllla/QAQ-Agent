# LangGraph + Electron 桌面端 AI Agent 开发文档

## 一、项目立项

### 1.1 项目概述

本项目旨在构建一个基于 **LangGraph** 的 **Windows 桌面端 AI Agent 应用**。该应用以 Electron 为桌面壳，集成 LangGraph.js 实现智能体推理循环，支持通过 OpenAI API 或本地 Ollama 模型驱动，具备工具调用（Tool Calling）能力，并以直观的聊天界面与用户交互。

### 1.2 项目目标

| 目标 | 说明 |
|------|------|
| 桌面端体验 | 原生 Windows 桌面应用，支持打包分发 |
| 智能 Agent | 基于 LangGraph 的 ReAct 模式，具备推理+工具调用能力 |
| 灵活 LLM 接入 | 支持 OpenAI API 和本地 Ollama 模型 |
| 可视化交互 | 现代化聊天界面，实时展示工具调用过程 |
| 可扩展架构 | 模块化设计，方便添加新工具和自定义 Agent 逻辑 |

## 五、核心实现详解

### 5.1 Electron Forge + Vite 配置

#### forge.config.js

```javascript
const { VitePlugin } = require('@electron-forge/plugin-vite');

module.exports = {
  packagerConfig: {
    asar: true,  // 将源码打包为 asar 归档，保护源码
  },
  makers: [
    {
      name: '@electron-forge/maker-squirrel',  // Windows 安装程序
      config: { name: 'langgraph_agent' },
    },
    {
      name: '@electron-forge/maker-zip',        // ZIP 压缩包
      platforms: ['win32'],
    },
  ],
  plugins: [
    new VitePlugin({
      build: [
        { entry: 'src/main.ts', config: 'vite.main.config.mjs', target: 'main' },
        { entry: 'src/preload.ts', config: 'vite.preload.config.mjs', target: 'preload' },
      ],
      renderer: [
        { name: 'main_window', config: 'vite.renderer.config.mjs' },
      ],
    }),
  ],
};
```

**要点说明：**
- `build` 数组定义主进程和 preload 的构建入口
- `renderer` 数组定义渲染进程的 Vite 开发服务器
- `asar: true` 在生产打包时将源码归档

#### vite.main.config.mjs (关键配置)

```javascript
import { defineConfig } from 'vite';

export default defineConfig({
  resolve: {
    conditions: ['node'],
    mainFields: ['module', 'jsnext:main', 'jsnext'],
  },
  build: {
    rollupOptions: {
      external: ['electron'],  // Electron 保持外部引用
    },
  },
  ssr: {
    // 将 ESM-only 的 LangChain 包内联打包，解决 CJS/ESM 兼容问题
    noExternal: [
      '@langchain/langgraph',
      '@langchain/core',
      '@langchain/openai',
      '@langchain/ollama',
      '@langchain/textsplitters',
      'zod',
    ],
  },
});
```

**关键决策：** `ssr.noExternal` 配置是本项目的一个技术难点。LangChain/LangGraph 的 npm 包发布为纯 ESM 格式，而 Electron 主进程默认使用 CJS 模块系统。通过 `noExternal` 将 LangChain 包内联到 Vite 的构建输出中，Vite 会自动将 ESM 转换为 CJS 兼容格式，解决了模块兼容问题。

### 5.2 Electron 主进程 (src/main.ts)

主进程是应用的核心枢纽，负责：
1. 创建和管理浏览器窗口
2. 处理 IPC 通信
3. 持久化用户设置
4. 调用 LangGraph Agent

```typescript
// 核心流程
app.whenReady().then(() => {
  createWindow();  // 创建主窗口
});

// IPC 处理 - Agent 对话
ipcMain.handle('agent:chat', async (_event, message: string) => {
  const results = await runAgent(message, currentSettings, (event) => {
    mainWindow?.webContents.send('agent:tool-event', event);  // 实时推送工具事件
  });
  return { success: true, data: results };
});

// IPC 处理 - 设置管理
ipcMain.handle('settings:get', () => currentSettings);
ipcMain.handle('settings:save', (_event, settings) => {
  currentSettings = settings;
  saveSettings(settings);  // 持久化到 userData 目录
});
```

**设置持久化策略：** 使用 JSON 文件存储在 Electron 的 `userData` 目录中（`%APPDATA%/langgraph-agent/agent-settings.json`），无需引入额外数据库依赖。

### 5.3 Preload 桥接 (src/preload.ts)

Preload 脚本在渲染进程加载前执行，通过 `contextBridge` 安全地暴露 Node.js API：

```typescript
import { contextBridge, ipcRenderer } from 'electron';

contextBridge.exposeInMainWorld('electronAPI', {
  chat: (message: string) => ipcRenderer.invoke('agent:chat', message),
  onToolEvent: (callback: (event: any) => void) => {
    const listener = (_event: any, data: any) => callback(data);
    ipcRenderer.on('agent:tool-event', listener);
    return () => ipcRenderer.removeListener('agent:tool-event', listener);
  },
  getSettings: () => ipcRenderer.invoke('settings:get'),
  saveSettings: (settings: any) => ipcRenderer.invoke('settings:save', settings),
});
```

**安全设计：**
- `contextIsolation: true` 确保渲染进程无法直接访问 Node.js
- 只暴露必要的 API 方法
- 使用 `invoke/handle` 模式实现请求-响应通信
- 使用 `on/removeListener` 实现事件订阅

### 5.4 LangGraph Agent 核心 (src/agent.ts)

这是项目最核心的部分，实现了基于 LangGraph 的 ReAct Agent。

#### 5.4.1 Agent 状态图

```
                    ┌──────────┐
         ┌─────────►│  Agent   │──── 无工具调用 ────► END
         │          │  (LLM)   │
         │          └──────────┘
         │               │
         │          有工具调用
         │               │
         │               ▼
         │          ┌──────────┐
         └──────────│  Tools   │
                    └──────────┘
```

Agent 遵循 ReAct (Reasoning + Acting) 模式：
1. **Agent 节点**：LLM 分析用户输入，决定是否调用工具
2. **条件路由**：检查 LLM 输出是否包含工具调用
3. **Tools 节点**：执行工具并将结果返回给 Agent
4. **循环直到**：LLM 认为已有足够信息，直接生成最终回答

#### 5.4.2 状态定义

```typescript
import { Annotation } from '@langchain/langgraph';
import type { BaseMessage } from '@langchain/core/messages';

const AgentState = Annotation.Root({
  messages: Annotation<BaseMessage[]>({
    reducer: (prev, next) => [...prev, ...next],  // 消息累加器
    default: () => [],
  }),
});
```

`Annotation` 是 LangGraph 的状态声明方式：
- `reducer` 定义状态更新策略：新消息追加到现有消息列表
- `default` 提供初始值

#### 5.4.3 工具定义

使用 LangChain 的 `tool()` 辅助函数和 Zod Schema 定义工具：

```typescript
import { tool } from '@langchain/core/tools';
import { z } from 'zod';

const calculatorTool = tool(
  async ({ expression }) => {
    const sanitized = expression.replace(/[^0-9+\-*/().%\s]/g, '');
    const result = new Function(`return (${sanitized})`)();
    return `计算结果: ${expression} = ${result}`;
  },
  {
    name: 'calculator',
    description: '执行数学计算，输入数学表达式',
    schema: z.object({
      expression: z.string().describe('要计算的数学表达式'),
    }),
  }
);
```

**当前内置工具：**

| 工具名 | 功能 | 输入参数 |
|--------|------|---------|
| `calculator` | 数学表达式求值 | `expression: string` |
| `get_datetime` | 获取当前日期时间 | `query: string` |
| `text_analysis` | 文本统计分析 | `text: string` |
| `random_number` | 生成随机整数 | `min: number, max: number` |

#### 5.4.4 图构建

```typescript
const graph = new StateGraph(AgentState)
  .addNode('agent', agentNode)           // 添加 Agent 节点
  .addNode('tools', toolsNode)           // 添加 Tools 节点
  .addEdge('__start__', 'agent')         // 入口 → Agent
  .addConditionalEdges('agent', shouldContinue, {
    tools: 'tools',                      // 有工具调用 → Tools
    end: END,                            // 无工具调用 → 结束
  })
  .addEdge('tools', 'agent')             // Tools → Agent (循环)
  .compile();                            // 编译为可执行图
```

#### 5.4.5 LLM 模型接入

```typescript
function createModel(settings: AgentSettings) {
  if (settings.provider === 'ollama') {
    return new ChatOllama({
      model: settings.model || 'llama3.1',
      baseUrl: settings.baseUrl || 'http://localhost:11434',
```

## 七、关键技术决策与挑战

### 7.1 ESM/CJS 模块兼容

**问题：** LangChain.js 生态发布为纯 ESM 包，而 Electron 主进程传统上使用 CJS。

**解决方案：** 在 `vite.main.config.mjs` 中使用 `ssr.noExternal` 将 LangChain 相关包内联打包。Vite 在打包时自动处理 ESM→CJS 转换。

```javascript
ssr: {
  noExternal: ['@langchain/langgraph', '@langchain/core', '@langchain/openai', ...]
}
```

### 7.2 进程间通信设计

**方案：** 采用 Electron 推荐的 `contextBridge` + `ipcMain.handle/ipcRenderer.invoke` 模式。

**优势：**
- 安全性：渲染进程完全隔离，无法直接访问 Node.js
- 类型安全：通过 TypeScript 接口定义通信契约
- 实时性：工具调用事件通过 `send/on` 单向推送

### 7.3 Agent 循环终止

**问题：** 如何确保 Agent 不会无限循环调用工具？

**方案：** LangGraph 的条件路由 `shouldContinue()` 检查 LLM 最新输出是否包含 `tool_calls`。当 LLM 认为信息足够时，不再发起工具调用，图自然终止。

### 8.4 流式输出

```typescript

const eventStream = await graph.streamEvents(
  { messages },
  { version: 'v2' }
);

for await (const event of eventStream) {
  if (event.event === 'on_chat_model_stream') {
    onStream?.(event.data.chunk.content);  // 逐字推送
  }
}
```

## 十、总结

4. **开发体验优化**：Vite 极速 HMR + TypeScript 类型安全 + 暗色主题 UI

项目从一个空目录开始，经过立项规划、技术选型、架构设计、编码实现到运行验证，完整展示了一个桌面端 AI Agent 应用从零到一的全过程。
